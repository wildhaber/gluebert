Underlaying Concepts

Gluebert is not Single-Page-Application Framework. If you are looking for an SPA-FW we recommend checking out something like vue.js, react, angular to name a few - they are great.

Great for classical architectures

With Gluebert we tried to solve a common issue we faced when it comes to the architectural decision for new Web-Application or regular Website Projects. Very often for several reasons we could not use a common SPA and end up with a classical approach either JAM-Stack of a server rendered solution e.g. Smyfony, Laravel, Wordpress or Drupal. Where one of the main challenge is not to mess up with DOM-Selectors.

In this case, Gluebert.js is your friend and binds the DOM element to the appropriate module whenever the element appears - no matter if the element existed initially or were created on runtime.

No payload and computation waste

The underlaying usage of MutationObserver and IntersectionObserver allows you to concentrate fully on your modules, styles and data binding. Furthermore all necessary dependencies are loaded lazy whenever actually needed, to optimize payload supporting first meaningful paint and benefit from HTTP2-concepts as well.

Currently we require a Webpack workflow that handles code-splitting allowing dynamic imports.
